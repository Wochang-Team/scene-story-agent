# 프로젝트 인사이트

이 문서는 `scene-story-agent` 개발 과정에서 얻은 공유 가능한 인사이트를 정리한다.

## LLM Wiki 운영

Git 저장소 안의 문서는 raw로 복사하지 않는다.

- 프로젝트 내부 문서는 원본 경로를 출처로 직접 연결한다.
- `.wiki/raw/`는 외부 원문, PDF, 회의록, 캡처처럼 Git 밖에서 들어온 자료만 보관한다.
- 작은 프로젝트 위키는 하위 디렉터리를 늘리기보다 `index`, `log`, `insights`를 먼저 잘 유지한다.
- 공식 문서로 검증한 결과는 원문 복사보다 확인일과 URL을 남기는 방식이 관리하기 쉽다.

## 제품 설계

이 제품의 기준 데이터는 원본 기록이다.

- AI 해석 정보는 원본 기록을 대체하지 않는다.
- 사용자가 남긴 사진, 영상, 메모, 감정, 만족도는 그대로 다시 볼 수 있어야 한다.
- AI가 만든 장소, OCR, 태그, 요약은 사용자가 확인하고 수정할 수 있어야 한다.
- 벡터 검색 결과는 사실이 아니라 후보로 취급한다.

## 데이터 경계

삭제는 원본 기록을 기준으로 묶어야 한다.

- 원본 파일, 사용자 입력값, AI 해석 정보, 임베딩, 연결 후보, 타임라인 반영 결과는 같은 삭제 경계에 둔다.
- 삭제 흐름은 API route에 흩어두지 않고 service 계층의 하나의 use case로 묶는다.
- 부분 실패가 가능한 Object Storage, PostgreSQL, pgvector 삭제는 보정 작업 기준을 둔다.

## AI Provider 선택

Provider 선택은 모델 품질보다 운영 조건까지 함께 본다.

- OpenAI는 Vision, LLM, Embedding을 한 공급자 안에서 구성하기 쉽다.
- Gemini는 개발·검증 비용을 낮추기 좋지만 Free Tier 데이터 사용 정책을 운영 기준으로 삼으면 안 된다.
- Provider별 가격, rate limit, 데이터 사용 정책은 출시 전 다시 확인한다.
- 모델명은 문서에 고정하되, 구현은 Provider interface 뒤에 숨긴다.

## 이미지 처리

이미지는 비용과 개인정보가 동시에 커지는 입력이다.

- OCR이 필요한 메뉴판, 영수증, 간판은 높은 detail을 검토한다.
- 음식, 공간, 분위기처럼 세밀한 글자 인식이 덜 중요한 이미지는 낮은 detail을 우선 검토한다.
- OpenAI 이미지 detail은 모델별 지원 범위가 다르므로 `gpt-5.4`와 `gpt-5.4-mini`를 구분한다.
- 업로드 시 EXIF 제거 여부는 개인정보 정책과 함께 결정한다.

## 인프라 설계

MVP 인프라는 관리형 서비스 조합이 맞다.

- API 서버는 Cloud Run 같은 컨테이너 실행 환경으로 시작한다.
- 짧은 비동기 요청은 Cloud Tasks, 길거나 병렬 처리가 필요한 작업은 Cloud Run Jobs를 검토한다.
- Supabase Postgres + pgvector는 MVP에서 정형 데이터와 벡터 데이터를 한 저장소에서 다루기 좋다.
- R2 presigned URL은 bearer token처럼 다루고 만료 시간을 짧게 둔다.
- Upstash Redis는 Redis 호환 서비스지만 사용할 명령은 호환성 확인이 필요하다.

## 개인정보

사진 기록 서비스는 기획 단계부터 개인정보 제품으로 봐야 한다.

- 사진, 영상, 메모, 장소 후보, 방문 시점, OCR, AI 요약, 임베딩은 모두 보호 대상이다.
- AI 모델 제공자에게 전송되는 데이터 항목은 처리방침에 공개한다.
- 국외 이전 또는 국외 처리위탁 여부는 Provider와 배포 리전을 정할 때 함께 확인한다.
- MVP에서 자동 인물 인식, 얼굴 식별, 실시간 위치 추적을 제외한 판단은 유지하는 편이 안전하다.
