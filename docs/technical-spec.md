# 기술 스펙

## 1. 문서 목적

이 문서는 `scene-story-agent`의 MVP 구현에 필요한 기술 선택 기준과 권장 기술을 정리한다.

프로젝트 개요, 사용자 시나리오, 핵심 데이터 모델은 README를 기준으로 한다. 이 문서는 구현 기술의 역할, 1순위 선택, 2순위 후보, 선택 이유만 다룬다.

## 2. 기술 선택 기준

기술은 다음 기준으로 판단한다.

- MVP 구현 속도
- AI 기능과의 연동성
- 운영 복잡도
- 데이터 구조와의 적합성
- 향후 확장 가능성
- 팀이 유지보수하기 쉬운 구조

## 3. 기술 선택 비교

| 영역 | 1순위 | 2순위 | 1순위 선택 이유 | 2순위인 이유 |
|---|---|---|---|---|
| API 서버 | FastAPI | Django REST Framework | Python 기반 AI 처리 코드와 잘 맞고, 비동기 API와 타입 기반 스키마 작성이 쉽다. | 관리자 기능과 ORM은 강하지만, MVP AI API 서버로는 구조가 무겁다. |
| 요청/응답 스키마 | Pydantic | Marshmallow | FastAPI와 기본 통합되고, 타입 힌트 기반 검증이 자연스럽다. | 독립 검증 도구로는 성숙하지만, FastAPI 조합에서는 통합성이 낮다. |
| 관계형 데이터베이스 | PostgreSQL | MySQL | JSON, 전문 검색, 확장 기능, pgvector 연동이 좋아 정형 데이터와 의미 검색을 함께 다루기 쉽다. | 안정적인 범용 DB지만, 벡터 검색 확장성과 AI 검색 조합은 PostgreSQL보다 약하다. |
| 벡터 검색 | pgvector | Chroma | PostgreSQL 안에서 구조화 데이터와 벡터 데이터를 함께 관리할 수 있어 MVP 운영이 단순하다. | 로컬 개발과 프로토타입에는 편하지만, 운영 DB와 분리되어 데이터 정합성 관리가 필요하다. |
| 원본 파일 저장 | S3 호환 스토리지 | 로컬 파일 시스템 | 이미지와 영상 파일 저장, 접근 권한 관리, 백업, 확장에 적합하다. | 개발 초기에는 간단하지만, 배포와 확장 환경에서는 한계가 크다. |
| Vision 모델 | OpenAI Vision | Google Cloud Vision | 장면 해석과 자연어 요약을 LLM 응답 생성 흐름과 함께 구성하기 쉽다. | OCR과 객체 인식은 강하지만, Story 생성까지 이어지는 자연어 맥락 처리는 별도 구성이 필요하다. |
| OCR | OpenAI Vision 기반 OCR | Google Cloud Vision OCR | 이미지 장면 해석과 텍스트 추출을 하나의 AI 흐름으로 처리할 수 있다. | OCR 정확도와 문서 처리 기능은 강하지만, 별도 API 연동과 결과 통합이 필요하다. |
| LLM | OpenAI GPT 계열 | Claude 계열 | Vision, 임베딩, 응답 생성을 한 공급자 안에서 구성하기 쉽다. | 긴 문맥 처리와 문서 작성에는 강점이 있지만, Vision과 임베딩까지 한 흐름으로 묶으려면 추가 설계가 필요하다. |
| 임베딩 | OpenAI Embeddings | Sentence Transformers | LLM, Vision 처리와 같은 공급자 안에서 의미 검색용 임베딩을 만들 수 있다. | 자체 운영과 비용 통제에는 장점이 있지만, 모델 선택과 서빙 운영 부담이 생긴다. |
| 에이전트 흐름 제어 | LangGraph | LangChain | 상태 기반 흐름, 조건 분기, 재시도, 단계별 처리를 명시적으로 구성하기 좋다. | 도구 생태계는 넓지만, 복잡한 상태 흐름 관리에서는 LangGraph보다 흐름이 덜 명확해질 수 있다. |
| 비동기 작업 | Celery | RQ | 이미지 분석, 임베딩 생성, Story 생성 같은 백그라운드 작업을 안정적으로 처리할 수 있다. | 구조가 단순하고 도입은 쉽지만, 복잡한 작업 흐름과 운영 기능은 Celery보다 제한적이다. |
| 캐시/큐 브로커 | Redis | RabbitMQ | Celery, RQ와 연동이 쉽고 캐시와 작업 큐 용도로 함께 쓸 수 있다. | 메시지 큐로는 강하지만, 캐시와 간단한 작업 큐까지 함께 쓰기에는 Redis가 더 단순하다. |
| ORM | SQLAlchemy | Django ORM | FastAPI와 조합하기 좋고, DB 모델을 독립적으로 설계하기 쉽다. | Django 안에서는 강력하지만, FastAPI 기반 프로젝트에서는 Django 구조를 함께 가져와야 한다. |
| 마이그레이션 | Alembic | Django Migration | SQLAlchemy와 함께 쓰기 좋고, FastAPI 프로젝트에서 표준적으로 사용된다. | Django ORM을 사용할 때는 편하지만, FastAPI와 SQLAlchemy 조합에는 맞지 않는다. |
| 테스트 | Pytest | unittest | FastAPI, 비동기 코드, fixture 기반 테스트 작성에 적합하다. | 표준 라이브러리라 의존성은 적지만, 복잡한 테스트 구성에서는 Pytest보다 생산성이 낮다. |
| API 문서 | FastAPI OpenAPI 자동 문서 | Redoc 별도 구성 | FastAPI가 기본 제공하므로 초기 API 확인과 테스트가 쉽다. | 문서 표현은 좋지만, 별도 커스터마이징과 관리가 필요하다. |
| 로컬 실행/배포 | Docker Compose | Kubernetes | PostgreSQL, Redis, API 서버, 워커를 MVP 환경에서 함께 띄우기 쉽다. | 확장성과 운영 자동화는 강하지만, MVP 단계에서는 관리 복잡도가 높다. |
| 설정 관리 | Pydantic Settings | python-dotenv | 타입 기반 환경 변수 검증이 가능하고, FastAPI/Pydantic 흐름과 맞다. | 단순 환경 변수 로딩에는 충분하지만, 설정 검증과 타입 관리 기능은 약하다. |
| 로그 | structlog | Python logging | 구조화 로그를 남기기 쉬워 API 요청, AI 호출, 백그라운드 작업 추적에 적합하다. | 표준 라이브러리라 도입은 쉽지만, 구조화 로그를 일관되게 관리하려면 추가 규칙이 필요하다. |

## 4. MVP 권장 조합

MVP에서는 다음 조합을 우선 적용한다.

- FastAPI
- Pydantic
- PostgreSQL
- pgvector
- S3 호환 스토리지
- OpenAI Vision
- OpenAI GPT 계열
- OpenAI Embeddings
- LangGraph
- Celery
- Redis
- SQLAlchemy
- Alembic
- Pytest
- Docker Compose
- Pydantic Settings
- structlog

이 조합은 Python 기반 AI 서버를 빠르게 만들 수 있고, 구조화 데이터와 벡터 검색을 한 저장소에서 관리할 수 있다. API 서버, 백그라운드 작업, DB, 큐를 로컬과 운영 환경에서 비슷한 구조로 실행할 수 있다.

## 5. AI 처리 구성요소 역할

AI 처리 흐름은 다음 구성요소로 나눈다.

| 구성요소 | 역할 | 하지 않는 일 |
|---|---|---|
| AI API | 사진과 텍스트를 이해하고 결과를 만든다. | 데이터를 장기 저장하거나 유사 기록을 직접 검색하지 않는다. |
| Embedding API | 기록의 의미를 숫자 벡터로 바꾼다. | 최종 판단이나 자연어 응답 생성을 하지 않는다. |
| Vector DB 또는 pgvector | 비슷한 기록을 빠르게 찾는다. | 왜 연결되는지 설명하거나 재방문 여부를 최종 판단하지 않는다. |
| PostgreSQL | 원본 기록, AI 해석 정보, 연결 결과를 저장한다. | 의미 기반 유사도 검색을 단독으로 처리하지 않는다. |
| Object Storage | 원본 사진과 영상을 저장한다. | 검색, 판단, 요약을 하지 않는다. |

## 6. 기본 AI 처리 흐름

기록 1건은 다음 순서로 처리한다.

1. 사용자가 사진을 업로드한다.
2. 원본 사진을 Object Storage에 저장한다.
3. 서버가 AI 처리용 리사이즈 이미지를 만든다.
4. AI API가 리사이즈 이미지에서 사진 정보를 추출한다.
5. 추출 결과를 PostgreSQL에 구조화 데이터로 저장한다.
6. 추출 결과 일부를 Embedding API에 전달한다.
7. Embedding API가 검색용 벡터를 생성한다.
8. 벡터를 Vector DB 또는 pgvector에 저장한다.
9. 새 기록의 벡터로 기존 기록을 검색한다.
10. Vector DB가 연관 기록 후보를 반환한다.
11. AI API가 현재 기록과 연관 기록 후보를 다시 분석한다.
12. AI API가 연결 후보, 재방문 판단, 타임라인 반영 결과를 생성한다.
13. PostgreSQL에 연결 결과를 저장한다.
14. 사용자 화면에 원본 기록, AI 해석 정보, 연결 후보를 함께 보여준다.

## 7. AI API 역할

AI API는 사진과 텍스트를 이해하고 사용자가 볼 수 있는 결과를 만든다.

### 입력

- 리사이즈 이미지
- 사용자 메모
- 감정
- 만족도
- 기록 작성 시점
- 연관 기록 후보

### 출력

- 장면 유형
- OCR 후보
- 장소 후보
- 메뉴 또는 활동 후보
- 금액 후보
- 태그
- 요약
- 연결 후보 판단
- 재방문 판단
- 타임라인 반영 결과

AI API는 두 번 호출될 수 있다.

1. 사진 정보 추출 호출
2. 현재 기록과 연관 기록 후보를 비교하는 결과 생성 호출

## 8. 임베딩 역할

임베딩은 기록의 의미를 검색 가능한 숫자 벡터로 바꾼다.

임베딩 입력은 원본 전체가 아니라 검색에 필요한 요약 정보로 만든다.

### 임베딩 입력 예시

```text
장소: 성수동 OO식당
장면: 영수증
메뉴: 돈카츠, 냉모밀
금액: 27000
태그: 일식, 점심, 성수
감정: 만족
요약: 성수동 OO식당에서 돈카츠와 냉모밀을 먹은 기록
```

### 임베딩 출력 예시

```json
[0.012, -0.238, 0.044]
```

임베딩은 다음 목적에 사용한다.

- 유사 기록 검색
- 같은 장소 후보 검색
- 비슷한 메뉴 또는 활동 검색
- 반복 주제 후보 검색
- 타임라인 연결 후보 검색

## 9. Vector DB 역할

Vector DB 또는 pgvector는 임베딩 벡터를 저장하고 비슷한 기록을 찾는다.

### 저장 데이터 예시

```json
{
  "record_id": "rec_123",
  "user_id": "user_1",
  "embedding": [0.012, -0.238, 0.044],
  "metadata": {
    "place_candidate": "성수동 OO식당",
    "scene_type": "restaurant_receipt",
    "happened_at": "2026-04-24",
    "tags": ["일식", "점심"]
  }
}
```

### 조회 결과 예시

```json
[
  {
    "record_id": "rec_123",
    "similarity": 0.91,
    "match_type": "same_place_candidate"
  },
  {
    "record_id": "rec_089",
    "similarity": 0.84,
    "match_type": "similar_topic"
  }
]
```

Vector DB 결과는 최종 판단이 아니라 후보로 사용한다. 최종 판단은 AI API가 현재 기록과 후보 기록을 함께 보고 생성한다.

## 10. PostgreSQL 역할

PostgreSQL은 서비스의 기준 데이터를 저장한다.

저장 대상은 다음이다.

- 사용자 계정
- 원본 기록 메타데이터
- 사진·영상 파일 URL
- 사용자 메모
- 감정
- 만족도
- AI 해석 정보
- 연결 후보
- 재방문 판단 결과
- 타임라인 반영 결과

pgvector를 PostgreSQL에 붙이면 구조화 데이터와 벡터 데이터를 같은 DB에서 관리할 수 있다. MVP에서는 운영 단순성을 위해 pgvector를 우선 사용한다.

## 11. 데이터 삭제 연계

사용자가 원본 기록을 삭제하면 다음 데이터를 함께 삭제한다.

- 원본 사진·영상 파일
- 사용자 입력값
- AI 해석·발췌 정보
- 임베딩 벡터
- 연결 후보
- 타임라인 반영 결과

삭제 처리는 PostgreSQL과 Vector DB 또는 pgvector, Object Storage가 같은 기준으로 동작해야 한다.

## 12. 2순위 기술 재검토 기준

2순위 기술은 다음 상황에서 다시 검토한다.

- 운영 요구가 바뀐다.
- 특정 기술의 비용이나 성능 문제가 커진다.
- 팀의 숙련도가 2순위 기술에 더 높다.
- 외부 시스템 연동 때문에 특정 기술을 써야 한다.
- 검색량, 이미지 처리량, 동시 작업량이 MVP 범위를 넘어선다.
- 관리형 서비스 도입으로 운영 책임을 줄여야 한다.

## 13. 초기 구현 순서

기술 도입은 다음 순서로 진행한다.

1. FastAPI, Pydantic, Pydantic Settings로 API 기본 구조를 만든다.
2. PostgreSQL, SQLAlchemy, Alembic으로 정형 데이터 저장 구조를 만든다.
3. Object Storage로 원본 파일 저장 경로를 분리한다.
4. AI API로 사진 정보 추출 흐름을 만든다.
5. Embedding API로 검색용 벡터 생성 흐름을 만든다.
6. pgvector로 기록 임베딩 저장과 유사 기록 검색 구조를 붙인다.
7. AI API로 현재 기록과 연관 기록 후보를 비교하는 결과 생성 흐름을 만든다.
8. LangGraph로 사진 정보 추출, 벡터 저장, 연관 데이터 조회, 결과 생성을 상태 흐름으로 구성한다.
9. Celery와 Redis로 이미지 분석, 임베딩 생성, 결과 생성을 비동기 처리한다.
10. Pytest로 API, 데이터 저장, 기억 조회, 삭제 연계를 검증한다.
11. Docker Compose로 로컬 실행 환경을 고정한다.

## 14. 운영 고려사항

초기 구현부터 다음 항목을 설정한다.

- 이미지 업로드 용량 제한
- 지원 이미지 형식 제한
- AI 호출 타임아웃

- 개인정보 마스킹 기준

- 임베딩 재생성 기준
